按需加载和文件合并的选择

发表于 2016-08-01   |   分类于 模块化   |  

关于前端模块化，一直是前端工程师所讨论的热点，在初期为了解决网站请求文件数量过多的问题，经常使用文件合并的方式，也就是通常所说的allinone，但这只是解决了网站的性能问题，而往往在大型的项目中，比如说多人合作的大型网站，经常会出现重复命名的方法，为此，我们可能会使用命名空间的方法，但这并不是解决问题的最好手段，模块化才是最终的解决方案。

早期为了实现前端模块化制定了amd和cmd两种规范，并由此诞生了requirejs和seajs两种工具，但这两种工具都有自身的缺陷。

  1. requirejs和seajs二者在加载上都有缺陷，就是模块的依赖要等到模块加载完成后，通过静态分析（seajs）或者deps参数（requirejs）来获取，这就为 合并请求 和 按需加载 带来了实现上的矛盾：

    • 要么放弃按需加载，把所有js合成一个文件，从而满足请求合并（两个框架的官方demo都有这样的例子）；
    • 要么放弃请求合并，请求独立的模块文件，从而满足按需加载。

  2. AMD规范在执行callback的时候，要初始化所有依赖的模块，而CMD只有执行到require的时候才初始化模块。所以用AMD实现某种if-else逻辑分支加载不同的模块的时候，就会比较麻烦了。考虑这种情况：

在执行回调之前，已经同时执行了index和detail模块的factory，而CMD只有执行到require才会调用对应模块的factory。这种差别带来的不仅仅是性能上的差异，也可能为开发增加一点小麻烦，比如不方便实现换肤功能，factory注意不要直接操作dom等。当然，我们可以多层嵌套require来解决这个问题，但又会引起模块请求串行的问题。

结论：以纯前端方式实现模块化框架 不能 同时满足 按需加载，请求合并 和 依赖管理 三个需求。导致这个问题的根本原因是 纯前端方式只能在运行时分析依赖关系,因此，对于网站来说，一般根据类型来选择合适的解决方案。
  1. 小型网站，不适用模块化方案，整体文件使用allinone方式。
  2. 中型网站，使用amd或者cmd规范进行开发，可以解决多人合作下的命名冲突问题。但是无法解决allinone问题。
  3. 大型网站，可以使用静态资源分析方式，通过和后台的配合对文件进行allinone，按需加载，做到网站性能的最大化，但是由于复杂度较高，所以一般只适用于大型网站。

在最新版本的js中，已经引入了前端模块化的解决方式，commonjs规范，由于目前浏览器端并不支持commonjs规范，所以可用性为零，但是在不久之后，当浏览器能够很好地支持es6之后，我们将无需因为模块化而设计如此多的解决方式。

网站图片懒加载技术的实现方式

发表于 2016-07-31   |   分类于 懒加载   |  

为了提高网站的加载速度，我们有一系列的方式来进行实现，其实我们知道影响加载速度的无非就是请求的数量和文件的大小，而请求数量的解决方式，之前我们通常会使用allinone的文件合并法和雪碧图的图标合并法来大量减少文件和图片的请求数量，这可能在很多类型的网站中都非常的实用，但还有一个非常重要的因素，就是显示区域问题。

假设一个非常庞大的电商网站，他的内容非常的多，我们如果一次性加载完成，那其实是非常消耗时间的，而且由于图片的加载属于异步加载，在网站图片过多的时候，会造成整个网站加载过程中的卡顿，对于这个问题，一般会使用两种方式进行解决。
  1. 一种是网站区域的整体懒加载（也就是说，一开始只加载人眼所能看到的区域内容，在滚动条下滑的过程中进行内容载入）。
  2. 第二种是使用图片懒加载技术，一开始只显示人眼所能识别区域的图片文件，根据滚动条的下拉来进行图片的载入，接下来我们具体讲解第二种实现方式。因为对于网站来说，文字的加载渲染并不需要太多的时间，影响网站性能最主要的因素还是图片。
具体实现图片懒加载技术并不复杂，其主要原理就是将图片的真是url地址放置在图片的自定属性之中，在需要其显示的时候将其url属性取出并复制给src，使得其去加载相应的图片，当然为了美观也可以在外层元素中加入背景图片属性

算法的复杂度和空间复杂度

发表于 2016-07-28   |  

算法效率的度量方法

  1. 事后统计方法，利用计时来进行计算（拥有很大的缺陷，必须依赖测试程序，花费太多时间，不同计算机处理速度不同）。
  2. 事前分析估算方法。受以下因素影响。
    1. 算法采用的策率
    2. 编译产生的代码质量
    3. 问题的输入规模
    4. 机器执行指令的速度

算法更需要关注主项的最高次方数

计算算法复杂度的攻略

计算算法复杂度的定义

一般情况下，算法中基本操作重复执行的次数是问题规模n的某个函数，用T(n)表示，若有某个辅助函数f(n),使得当n趋近于无穷大时，T(n)/f(n)的极限值为不等于零的常数，则称f(n)是T(n)的同数量级函数。记作T(n)=O(f(n)),称O(f(n)) 为算法的渐进时间复杂度，简称时间复杂度。
分析：随着模块n的增大，算法执行的时间的增长率和 f(n) 的增长率成正比，所以 f(n) 越小，算法的时间复杂度越低，算法的效率越高。

如何分析算法的复杂度

  1. 用常数1取代运行时间中的所有加法常数
  2. 在修改后的运行次数函数中，只保留最高项
  3. 最高项存在阶项且不为1，取消与其相乘的常熟

数据结构和算法绪论

发表于 2016-07-26   |   分类于 算法   |  

数据结构

程序设计=数据结构+算法。

数据结构就是关系，是数据元素相互之间存在的一种联系。
  • 集合结构：数据元素同属于一个集合，无其他联系。
  • 线性结构：元素属于一对一的结构。
  • 树形结构：元素存在一对多的形式。
  • 图形结构：多对多的数据结构。
  • 存储结构，分顺序存储结构和链式存储结构，顺序存储结构例如数组，链式存储结构，只需要关注前一个数据是否存储，元素可以是不连续的存储。以指针链接。
算法的特性
  1. 输入：算法具有零个或者多个输入
  2. 输出：算法至少有一个或者多个输出
  3. 有穷性，算法在执行有限步骤后会自动结束。
  4. 确定性：算法的每一个步骤都有明确的含义，相同的输入只会有唯一的的输出结果（有异议）。
  5. 可行性：算法的每一步在当前的环境下都是可行的
算法设计的要求
  1. 正确性：大体分为4个层次
    • 算法程序没有语法错误。
    • 算法程序对于合法输入能够有满足要求的输出。
    • 算法对于非法输入能够有满足规格的说明（给予提示）。
    • 算法程序对于故意刁难的测试输入有满足要求的输出结果。
  2. 可读性
    • 可读性，足够的注释和说明。
    • 健壮性，当输入不合法的数据时，算法应该给予相关处理，而不是产生异常，奔溃，或者莫名其妙的结果。
    • 时间效率高和存储量低。
